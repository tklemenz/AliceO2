1 # Data Processing Layer in O2 Framework
2 
3 ## Status quo and motivation for an O2 Data Processing Layer
4 
5 FairMQ currently provides a well documented and flexible framework for an
6 actor based computation where each of the actors listens for message-like
7 entities on channels and executes some code as a reaction. The key component
8 which controls this is called a `FairMQDevice` (or *device* from now on) which
9 can use different kind of transports to receive and send messages. In the most
10 generic case, users are allowed to have full control on the state machine
11 governing the message passing and have complete control on how the message
12 handling is done. This of course covers all ALICE usecases in a generic
13 manner, at the cost of extra complexity left to the user to implement. In most
14 cases however a simplified way of creating devices is provided, and the user
15 will simply create its own `FairMQDevice`-derived class, which registers via
16 the `OnData(FairMQParts &parts)` method a callback that is invoked whenever a
17 new message arrives. This however still holds the user responsible for:
18 
19 - Verifying that the required inputs for the computation are all available,
20  both from the actual data flow (being it for readout, reconstruction
21  or analysis) and from the asynchronous stream (e.g. alignment and
22  calibrations).
23 - Create the appropriate message which holds the results and send it.
24 - Ensure the ease of testability, the code reuse and the proper documentation
25  `OnData` callback. In particular there is no way to inspect which data is
26  expected by a device and which data is produced.
27 
28 This is by design, because the FairMQ transport layer should not know anything
29 about the actual data being transferred, while all the points above require
30 some sort of inner knowledge about the data model and the data being moved
31 around.
32 
33 The aim is to achieve the following:
34 
35 - **Explicit data flow:** Input and outputs are declared upfront and can be
36  used for documentation or for automatic topology creation (assuming the
37  actual processing environment is known).
38 - **Transport agnostic data processing:** users will not have to know about
39  the details of how the data materialises on their device, they will always
40  be handed the full set of payloads they requested, even if they come at
41  different time.
42 - **Composability of data processing:** different process functions can in
43  principle be chained and scheduled together depending on the desired
44  granularity for devices.
45 
46 ## Separating data-processing from transport
47 
48 For the reasons mentioned above, we propose that the one of the developments
49 which should happen with the O2 Framework work package is the development of a
50 “Data Processing layer” which actually knows about the O2 Data Model (and for
51 this reason cannot be part of FairMQ itself) and exploits it to validate,
52 optimise and correctly schedule a computation on a user specified set of
53 inputs.
54 
55 The Data Processing Layer in particular requires:
56 
57 - That the inputs of each computation are provided upfront.
58 - That the outputs of each computation are provided upfront.
59 - That a time identifier can be associated to inputs
60 
61 and given these premises it actually guarantees:
62 
63 - That no computation is performed before all the inputs for a given time
64  identifier are available
65 - That no message passing happens during the performing of the computation,
66  but uniquely at the end.
67 
68 ### Instanciating a computation
69 
70 The description of the computation in such the Data Processing Layer is done
71 via instances of the [`DataProcessorSpec`][DataProcessorSpec] class, grouped
72 in a so called `WorkflowSpec` instance. In order to provide a description a
73 computation to be run, the user must implement a callback which modifies an
74 empty `WorkflowSpec` instance provided by the system. E.g.:
75 
76 
77 ~~~~~~~{.cpp}
78 
79 #include "Framework/Utils/runDataProcessing.h"
80 
81 void defineDataProcessing(WorkflowSpec &workflow) {
82  auto spec = DataProcessorSpec{
83  ...
84  };
85  // Fill a DataProcessingSpec "spec"
86  workflow.push_back(spec);
87 }
88 ~~~~~~~
89 
90 See next section, for a more detailed description of the
91 [`DataProcessorSpec`][DataProcessorSpec] class. The code above has to be
92 linked into a single executable together with the Data Processing Layer code
93 to form a so called driver executable which if run will:
94 
95 - Map all [`DataProcessorSpec`][DataProcessorSpec] to a set of `FairMQDevice`s
96  (using 1-1 correspondence, in the current implementation).
97 - Instanciate and start all the devices resulted from the previous step.
98 - (Optionally) start a GUI which allows to monitor the running of the system.
99 
100 [DataProcessorSpec]: https://github.com/AliceO2Group/AliceO2/blob/dev/Framework/Core/include/Framework/DataProcessorSpec.h

102 ### Describing a computation
103 
104 The description of the computation in such a layer is done via a
105 `DataProcessorSpec` class, which describes some sort of processing of a (set
106 of) O2 Data Payloads (*payloads* from now on), as defined by the O2 Data
107 Model, eventually producing new payloads as outputs. The inputs to the
108 computation, the outputs and the actual code to run on the former to
109 produce the latter, is specified in a `DataProcessorSpec` instance. Multiple
110 `DataProcessorSpec` instances can be grouped together in a `WorkflowSpec`. to
111 the driver code which maps configures the processing device accordingly.
112 Practically speaking this would translate into something similar to the
113 current “simplified device mode” where the user includes a special header
114 which contains all the boilerplate and provides a well defined callback where
115 the requested `DataProcessorSpec` are provided.
116 
117 The `DataProcessorSpec` is defined as follows:
118 
119 
120 ~~~~~~~{.cpp}
121 
122 struct DataProcessorSpec {
123  using InitCallback = std::function<ProcessCallback(InitContext &)>;
124  using ProcessCallback = std::function<void(ProcessingContext &)>;
125  using ErrorCallback = std::function<void(ErrorContext &)>;
126  std::vector<InputSpec> inputs;
127  std::vector<OutputSpec> outputs;
128  std::vector<ConfigParamSpec> configParams;
129  std::vector<std::string> requiredServices;
130  AlgorithmSpec algorithm;
131 };
132 ~~~~~~~
133 
134 In the above both `InputSpec` and `OutputSpec` are like:
135 
136 
137 ~~~~~~~{.cpp}
138 
139 struct InputSpec { // OutputSpec as well
140  o2::Headers::DataDescription description;
141  o2::Headers::DataOrigin origin;
142  o2::Headers::SubSpecificationType subSpec;
143  enum Lifetime lifetime;
144 };
145 ~~~~~~~
146 
147 where description, origin and subSpec match the O2 Data Model definition.
148 For the moment we will consider this a one to one mapping with the
149 `o2::Headers::DataHeader` ones. In principle one could think of a one-to-many
150 relationship (e.g. give me all the clusters, regardless of their provenance)
151 and the processing layer could automatically aggregate those in a unique view.
152 This is also the semantic difference between `InputSpec` and `OutputSpec`: the
153 former is to express data that matches a given query (which must be exact at
154 the moment) the latter is to describe in all details and without any doubt the
155 kind of the produced outputs.
156 
157 The `lifetime` property:
158 
159 
160 ~~~~~~~{.cpp}
161 
162 enum Lifetime {
163  Timeframe,
164  Condition,
165  QA,
166  Transient
167 };
168 ~~~~~~~
169 
170 can be used to distinguish if the associated payload should be considered
171 payload data, and therefore be processed only once, or alignment / conditions
172 data, and therefore it would be considered valid until a new copy is made
173 available to the device.
174 
175 The `configParams` vector would be used to specify which configuration options
176 the data processing being described requires:
177 
178 
179 ~~~~~~~{.cpp}
180 
181 struct ConfigParamSpec {
182  std::string name;
183  enum ParamType type;
184  variant defaultValue;
185 };
186 ~~~~~~~
187 
188 command line / configuration options would be automatically generated by it.
189 These are available only at init stage, and can be used to configure services.
190 They are not available to the actual `process` callback as all the critical
191 parameters for data processing should be part of the data stream itself,
192 eventually coming from CCDB / ParameterManager.
193 
194 Similarly the `requiredServices` vector would define which services are
195 required for the data processing. For example this could be used to declare
196 the need for some data cache, a GPU context, a thread pool.
197 
198 The `algorithm` property, of `AlgorithmSpec` is instead used to specify
199 the actual computation. Notice that the same `DataProcessorSpec` can used
200 different `AlgorithmSpec`. The rationale for this is that while inputs and
201 outputs might be the same, you might want to compare different versions of
202 your algorithm. The `AlgorithmSpec` resembles the following:
203 
204 
205 ~~~~~~~{.cpp}
206 
207 struct AlgorithmSpec {
208  using ProcessCallback = std::function<void(ProcessingContext &)>;
209  using InitCallback = std::function<ProcessCallback(InitContext &)>;
210  using ErrorCallback = std::function<void(ErrorContext &)>;
211 
212  InitCallback onInit = nullptr;
213  ProcessCallback onProcess = nullptr;
214  ErrorCallback onError = nullptr;
215 };
216 ~~~~~~~
217 
218 The `onProcess` function is to be used for stateless computations. It’s a free
219 function and it’s up to the framework to make sure that all the required
220 components are declared upfront. It takes as input the context for the current
221 computation in the form of a `ProcessingContext &` instance. Such a context
222 consist of:
223 
224 - An `InputRecord` which allows retrieving the current inputs matching the provided
225  specification.
226 - A `ServiceRegistry` referencing the set of services it declared as required
227  the computation.
228 - A `DataAllocator` allocator which can allocate new payloads only for the
229  types which have been declared as `outputs`.
230 
231 `onProcess` is useful whenever your computation is fully contained in your
232 input. In several cases, however, a computation requires some ancillary state,
233 which needs to be initialised only on (re-)start of the job. For example you
234 might want to initialise the geometry of your detector. To do so, you can use
235 the `onInit` callback and allocate the state and pass it to the returned
236 `ProcessCallback` as captured arguments. E.g:
237 
238 
239 ~~~~~~~{.cpp}
240 
241 AlgorithmSpec{
242  InitCallBack{[](InitContext &setup){
243  auto statefulGeo = std::make_shared<TGeo>();
244  return [geo = statefulGeo](ProcessingContext &) {
245  // do something with geo
246  };
247  }
248  }
249 }
250 ~~~~~~~
251 
252 A `DataRef` would look like:
253 
254 
255 ~~~~~~~{.cpp}
256 
257 struct DataRef {
258  const InputSpec *spec;
259  const char *const header;
260  const char *const payload;
261 };
262 ~~~~~~~
263 
264 `header` and `payload` are the pointers to the data which matches the `spec`
265 InputSpec.

267 ## Implementing a computation
268 
269 This chapter describes how to actually implement an `AlgorithmSpec`.
270 
271 ### Using inputs - the `InputRecord` API
272 
273 Inputs to your computation will be provided to you via the
274 [`InputRecord`][InputRecord] API. An instance of such a class is hanging from
275 the `ProcessingContext` your computation lambda is passed and contains one
276 value for each of the `InputSpec` you specified. E.g.:
277 
278 
279 ~~~~~~~{.cpp}
280 
281 InputRecord &args = ctx.inputs();
282 ~~~~~~~
283 
284 From the `InputRecord` instance you can get the arguments either via their positional
285 index:
286 
287 
288 ~~~~~~~{.cpp}
289 
290 DataRef ref = args.getByPos(0);
291 ~~~~~~~
292 
293 or using the mnemonics-label which was used as first argument in the associated
294 `InputSpec`.
295 
296 
297 ~~~~~~~{.cpp}
298 
299 DataRef ref = args.get("points");
300 ~~~~~~~
301 
302 You can then use the `DataRef` `header` and `payload` raw pointers to access
303 the data in the messages.
304 
305 If the message is of a known type, you can automatically get a casted reference
306 to the contents of the message by passing it as template argument, e.g.:
307 
308 
309 ~~~~~~~{.cpp}
310 
311 XYZ &p = args.get<XYZ>("points");
312 ~~~~~~~
313 
314 [InputRecord]: https://github.com/AliceO2Group/AliceO2/blob/HEAD/Framework/Core/include/Framework/InputRecord.h
315 
316 ### Creating outputs - the DataAllocator API
317 
318 In order to prevent algorithms to create data they are not supposed to
319 create, a special `DataAllocator` object is passed to the process callback, so
320 that only messages for declared outputs can be created. A `DataAllocator` can
321 create Framework owned resources via the `make<T>` method. In case you ask the
322 framework to create a collection of objects, the result will be a `gsl::span`
323 wrapper around the collection. A `DataAllocator` can adopt externally created
324 resources via the `adopt` method. A `DataAllocator` can create a copy of an
325 externally owned resource via the `snapshot` method.
326 
327 Currently supported data types are:
328 
329 - Vanilla `char *` buffers with associated size. This is the actual contents of
330  the FairMQ message.
331 - POD types. These get directly mapped on the message exchanged by FairMQ and
332  are therefore "zerocopy" for what the DataProcessingLayer is concerned.
333 - POD collections, which are exposed to the user as `gsl::span`.
334 - TObject derived classes. These are actually serialised via a TMessage
335  and therefore are only suitable for the cases in which the cost of such a
336  serialization is not an issue.
337 
338 Currently supported data types for snapshot functionality, the state at time of
339 calling snapshot is captured in a copy:
340 - POD types
341 - TObject derived classes, serialized
342 - std::vector of POD type, at the receiver side the collection is exposed
343  as gsl::span
344 - std::vector pointer to POD type, the objects are linearized in the message
345  and exposed as gsl::span on the receiver side
346 
347 The DataChunk object resembles a `iovec`:
348 
349 
350 ~~~~~~~{.cpp}
351 
352 struct DataChunk {
353  char *data;
354  size_t size;
355 };
356 ~~~~~~~
357 
358 however, no API is provided to explicitly send it. All the created DataChunks
359 are sent (potentially using scatter / gather) when the `process` function
360 returns. This is to avoid the “modified after send” issues where a message
361 which was sent is still owned and modifiable by the creator.
362 
363 ### Error handling
364 
365 When an error happens during processing of some data, the writer of the
366 `process` function should simply throw an exception. By default the exception
367 is caught by the `DataProcessorManager` and a message is printed (if
368 `std::exeception` derived `what()` method is used, otherwise a generic message
369 is given). Users can provide themselves an error handler by specifying via the
370 `onError` callback specified in `DataProcessorSpec`.
371 
372 ### Services
373 
374 Services are utility classes which `DataProcessor`s can access to request
375 out-of-bound, deployment dependent, functionalities. For example a service
376 could be used to post metrics to the monitoring system or to get a GPU
377 context. The former would be dependent on whether you are running on your
378 laptop (where monitoring could simply mean print out metrics on the command
379 line) or in a large cluster (where monitoring probably means to send metrics
380 to an aggregator device which then pushes them to the backend.
381 
382 Services are initialised by the driver code (i.e. the code included via
383 runDataProcessing.h) and passed to the user code via a `ServiceRegistry`. You
384 can retrieve the service by the type of its interface class. E.g. for
385 monitoring you can do:
386 
387 
388 ~~~~~~~{.cpp}
389 
390 #include "Framework/MetricsService.h"
391 // ...
392 auto service = ctx.services().get<MetricsService>(); // In the DataProcessor lambda...
393 service.post("my/metric", 1); ...
394 ~~~~~~~
395 
396 Currently available services are described below.
397 
398 #### ControlService
399 
400 The control service allow DataProcessors to modify their state or the one of
401 their peers in the topology. For example if you want to quit the whole data
402 processing topology, you can use:
403 
404 
405 ~~~~~~~{.cpp}
406 
407 #include "Framework/ControlService.h"
408 //...
409 auto ctx.services().get<ControlService>().readyToQuit(true) // In the DataProcessor lambda
410 ~~~~~~~
411 
412 #### RawDeviceService
413 
414 This service allows you to get an hold of the `FairMQDevice` running the
415 DataProcessor computation from with the computation itself. While in general
416 this should not be used, it is handy in case you want to integrate with a
417 pre-existing `FairMQDevice` which potentially does not even follow the O2 Data
418 Model.

422 ### Debugging on your laptop
423 
424 The way the DPL currently works is that the driver executable you launch,
425 will then take care of spawning one device per `DataProcessorSpec` in
426 a separate process. This means that in order to debug your code you need to
427 make sure gdb / lldb are actually debugging the right child process.
428 
429 For `gdb` you can use the `follow-fork-mode` setting. See
430 [here](https://sourceware.org/gdb/onlinedocs/gdb/Forks.html) for the full
431 documentation. This is unfortunately not available in
432 [lldb](https://bugs.llvm.org/show_bug.cgi?id=17972).
433 
434 Alternatively you can start your driver executable with the `-s` / `--stop`
435 command line option which will immediately stop execution of the children after
436 the fork, allowing you to attach to them, e.g. for gdb using:
437 
438 
439 ~~~~~~~{.sh}
440 
441 attach <pid>
442 ~~~~~~~
443 
444 or the `lldb` equivalent:
445 
446 
447 ~~~~~~~{.sh}
448 
449 attach -pid <pid>
450 ~~~~~~~

452 ### Expressing parallelism
453 
454 If we want to retain a message passing semantic and really treat shared memory
455 as yet another transport, we need to be very careful in how to express
456 parallelism on data, so that the “single ownership model” of message passing
457 forces us to either duplicate streams that need to be accessed in parallel, or
458 to serialise workers which need to access the same data. Solutions like
459 reference counting shared memory would not be allowed in such a scenario and
460 in any case would require extra caution and support to make sure that failures
461 do not leave dangling reference around (e.g. when one of the parallel workers
462 abruptly terminates). First of all let’s consider the fact that there are two
463 level of parallelisms which can be achieved:
464 
465 - Data flow parallelism: when data can be split in partitions according to
466  some subdivision criteria (e.g. have one stream per TPC sector and have one
467  worker for each).
468 - Time flow parallelism: when parallelism can be achieved by having different
469  workers handle different time intervals for the incoming data. (e.g. worker 0
470  processes even timeframes, worker 1 processes odd timeframes).
471 
472 Data flow parallelism is simply expressed by tuning the data flow, adding
473 explicitly the parallel data paths, using the appropriate `InputSpec` and
474 `OutputSpec`. E.g.:
475 
476 
477 ~~~~~~~{.cpp}
478 
479 DataProcessorSpec{
480  "tpc_processor_1",
481  Inputs{},
482  Outputs{{"TPC", "CLUSTERS", SubSpec(0)}},
483  // ...
484 },
485 DataProcessorSpec{
486  "tpc_processor_2",
487  Inputs{},
488  Outputs{{"TPC", "CLUSTERS", SubSpec(1)}},
489  // ...
490 }
491 // ...
492 DataProcessorSpec{
493  "tpc_processor_18",
494  Inputs{},
495  Outputs{{"TPC", "CLUSTERS", SubSpec(17)}},
496  // ...
497 }
498 ~~~~~~~
499 
500 or alternatively the parallel workflows part could be generated
501 programmatically:
502 
503 
504 ~~~~~~~{.cpp}
505 
506 parallel(
507  DataProcessorSpec{
508  "tpc_processor",
509  {InputSpec{"c", "TPC", "CLUSTERS"}}
510  },
511  18,
512  [](DataProcessorSpec &spec, size_t idx) {
513  spec.outputs[0].subSpec = idx; // Each of the 18 DataProcessorSpecs should have a different subSpec
514  }
515 )
516 ~~~~~~~
517 
518 Similarly this can be done for a component that merges inputs from multiple
519 parallel devices, this time by modifying programmatically the `Inputs`:
520 
521 
522 ~~~~~~~{.cpp}
523 
524 // ...
525 DataProcessorSpec{
526  "merger",
527  mergeInputs({"a", "TST", "A", InputSpec::Timeframe},
528  4,
529  [](InputSpec &input, size_t index) {
530  input.subSpec = index;
531  }
532  ),
533  {},
534  AlgorithmSpec{[](InitContext &setup) {
535  return [](ProcessingContext &ctx) {
536  // Create a single output.
537  LOG(DEBUG) << "Invoked" << std::endl;
538  };
539 }
540 // ...
541 ~~~~~~~
542 
543 When one declares a parallel set of devices you can retrieve the rank (i.e.
544 parallel id) or the number of parallel devices by using the `ParalleContext`,
545 which can be retrieved from the `ServiceRegistry` (see also the `Services`
546 section below), e.g.:
547 
548  size_t whoAmI = services.get<ParallelContext>().index1D();
549  size_t howManyAreWe = services.get<ParallelContext>().index1DSize();
550 
551 A second type of parallelism is time based pipelining. This assumes that the
552 data can be subdivided in subsequent "time periods" that are independent one
553 from the other and which are each identified by some timestamp entity. In this
554 particular case it could result handy that some part of the workflow are
555 actually processing different time periods. This can be expressed via the
556 `timePipeline`, directive, e.g.:
557 
558 
559 ~~~~~~~{.cpp}
560 
561 // ...
562 timePipeline(DataProcessorSpec{
563  "processor",
564  {InputSpec{"a", "TST", "A"}},
565  {OutputSpec{"TST", "B"}},
566  AlgorithmSpec{[](ProcessingContext &ctx) {
567  };
568  }
569 }, 2);
570 // ...
571 ~~~~~~~
572 
573 which will result in two devices, one for even time periods, the other one for
574 odd timeperiods.

590 ### Integrating with pre-existing devices
591 
592 Given the Data Processing Layer comes somewhat later in the design of O2, it's
593 possible that you already have some topology of devices which you want to
594 integrate, without having to port them to the DPL itself. Alternatively, your
595 devices might not satisfy the requirements of the Data Processing Layer and
596 therefore require a "raw" `FairMQDevice`, fully customised to your needs.
597 This is fully supported and we provide means to ingest foreign, non-DPL
598 FairMQDevices produced, messages into a DPL workflow. This is done via the
599 help of a "proxy" data processor which connects to the foreign device, receives its
600 inputs, optionally converts them to a format understood by the Data Processing
601 Layer, and then pumps them to the right Data Processor Specs. In order to have
602 such a device in your workflow, you can use the
603 [`specifyExternalFairMQDeviceProxy`][specifyExternalFairMQDeviceProxy] helper to instanciate it.
604 For an example of how to use it you can look at
605 [`Framework/TestWorkflows/src/test_RawDeviceInjector.cxx`][rawDeviceInjectorExample].
606 The `specifyExternalFairMQDeviceProxy` takes four arguments:
607 
608 
609 ~~~~~~~{.cpp}
610 
611 specifyExternalFairMQDeviceProxy("foreign-source",
612  {outspec},
613  "type=sub,method=connect,address=tcp://localhost:5450,rateLogging=1",
614  o2DataModelAdaptor(outspec, 0, 1)
615  ),
616 ~~~~~~~
617 
618 the first one is the usual `DataProcessorSpec` name, the second one is a list
619 of outputs which we will create from the non-DPL device, the third one is a
620 string to connect to the existing topology and a the fourth one is a function
621 of the kind `o2::framework::InjectorFunction` which does the actual conversion.
622 In this particular case we use the `o2DataModelAdaptor()` helper to create such an
623 translation function since we know that our input is already respecting the O2
624 Data Model and most of the heavylifing can be done automatically.
625 
626 Sending out the results of a computation can be done in a similar manner. Use
627 
628 
629 ~~~~~~~{.cpp}
630 
631 ConfigParamSpec{"channel-config", VariantType::String, "<channel-configuration>", "Out-of-band channel config"}
632 ~~~~~~~
633 
634 to create an out-of-band channel as specified in `channel-configuration` and
635 then use the `RawDeviceService` to get the raw FairMQDevice and send data
636 through such a channel.
