4 ## Instructions for the contributors
5 
6 A sub-module CMakeLists.txt minimally contains (see Examples/ExampleModule1)
7 
8 * The setup the system : `O2_SETUP(NAME My_Module)`
9 * The list of the source files in the variable SRC : `set(SRCS something.cxx)`
10 * The name of the library : `set(LIBRARY_NAME My_Module)`
11 * The name of the dependency bucket to use : `set(BUCKET_NAME My_bucket)`
12 * The call to generate the library : `O2_GENERATE_LIBRARY()`
13 
14 Optionally it contains (see Examples/ExampleModule2)
15 
16 * To generate a dictionary :
17  * The list of source files not to be used for the dictionary : `set(NO_DICT_SRCS src/Bar.cxx)`
18  * The list of headers (private and public) : `set(HEADERS include/${MODULE_NAME}/Foo.h)`
19  * The linkdef : `set(LINKDEF src/ExampleLinkDef.h)`
20 * To generate an executable :
21  ```
22  O2_GENERATE_EXECUTABLE(
23  EXE_NAME runExampleModule1
24  SOURCES src/main.cxx
25  MODULE_LIBRARY_NAME ${LIBRARY_NAME}
26  BUCKET_NAME ${BUCKET_NAME}
27  )
28  ```
29 
30 If a new bucket is needed, it should be defined in cmake/O2Dependencies.cmake using this function :
31 ```
32 o2_define_bucket(
33  NAME
34  ExampleModule2_bucket
35 
36  DEPENDENCIES
37  ${Boost_PROGRAM_OPTIONS_LIBRARY} # a library
38  ExampleModule1 # another module
39  ExampleModule1_bucket # another bucket
40 
41  INCLUDE_DIRECTORIES
42  ${CMAKE_SOURCE_DIR}/Examples/ExampleModule1/include # another module's include dir
43 
44  SYSTEMINCLUDE_DIRECTORIES
45  ${Boost_INCLUDE_DIR} # a system lib include dir
46 )
47 ```
48 
49 If it needs a new external library, it should be first discussed with WP3.

51 ## Developers' documentation
52 
53 * Q: Why are the libraries' directories globally set in O2Dependencies.cmake ?
54  * A: CMake discourages the use of _link_directories_ because find_package and find_library
55  should return absolute paths. As a consequence little effort is put in the development of this
56  feature and it only exists at the global level. We can't set it on a target like the
57  _include_directories_ for example.
58 * Q: Why buckets ?
59  * A: The goal is to avoid a dependency nightmare.
60  It allows to define centrally and in an organized way the dependencies for all modules.
61  It also allows us to be especially careful in PRs about changes to the bucket definition file.
62 * Q: Why macros to build libraries and executables ?
63  * A: To simplify the life of the users and to make sure everyone does it the same way. It is also a way
64  to reuse what was made for FairRoot.
